(WEB) JWT token 인증 도입하기 (2)

posted by choieastsea,
October 06, 2023

오늘은 jwt token을 구현해보는 날이다. 서버와 클라이언트 모두 구현하므로, 개념만 챙겨가도 좋을 것이다!

우선 서버부터 구현하는 것이 편하다. fast api에서는 /docs url을 통한 swagger ui로 거의 모든 테스트가 가능하므로 편하다.

참고로 fast api서버는 8000번, react는 3000번 포트에서 테스트하였다.

Fastapi에서 해줘야 할 것들

공식문서가 매우 친절한 fast api에 OAuth2.0 인증과 관련한 문서와 모듈들이 제공된다. 공식문서에서는 access token만 있어 이를 응용하여 refresh token을 만들어볼 수 있다.

추가적으로, refresh token은 response header를 통하여 set-cookie로 전달하며, access token 은 response body를 통하여 bearer 토큰을 SPA의 variable로 갖고 있을 수 있도록 전달해주면 된다.

옵션 (CORS & credentials)

# main.py
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

클라이언트는 3000, 서버는 8000번 포트에서 서빙되므로 CROSS ORIGIN이다. 이에 대한 설정을 해줄 필요가 있다.

  • refresh token은 set-cookie로 넣어줄 것이므로, cross origin에서 set-cookie를 하고 이후에 cookie 값을 받아오려면 allow_credentials 설정을 활성화해야한다. 클라이언트는 credential 정보를 include한 AJAX 요청을 보내야 할 것이다.
  • allow_credential 옵션을 활성화하였다면 allow_origins를 ”*“가 아닌 정해진 주소들의 배열로 정해줘야 할 것이다.

토큰 정보 및 생성

이제 토큰을 위한 상수를 선언하자. 우선, access token과 refresh token 모두 JWT로 만들 것이고, 각각 다른 secret key를 이용하는 것이 권장된다. 아래의 키는 노출되는 순간 보안은 끝이므로, 조심히 보관하도록 하자.

# auth.py
from typing import Final

ACCESS_TOKEN_SECRET_KEY : Final[str] = "7f13039fb7c2d98fbbc8440794c2b6c006e066909190c3d055f7c13475e5a052"
REFRESH_TOKEN_SECRET_KEY : Final[str] = "3b78bb05f22b376a2938a9483afccb5f5797b8611ede32004e735c19e6056245"

ALGORITHM : Final[str] = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES : Final[int] = 30
REFRESH_TOKEN_EXPIRE_MINUTES : Final[int] = 60 * 24 * 7

이제 토큰을 생성하는 로직을 짜보자.

# auth.py
from datetime import datetime, timedelta
from typing import Union
from jose import jwt

def create_access_token(data: dict):
    """
    data : payload dictionary
    """
    to_encode = data.copy()
    expire = datetime.utcnow() + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, ACCESS_TOKEN_SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt

def create_refresh_token(data:dict):
    """
    data : payload dictionary
    """
    to_encode = data.copy()
    expire = timedelta(minutes=REFRESH_TOKEN_EXPIRE_MINUTES) + datetime.utcnow()
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(to_encode, REFRESH_TOKEN_SECRET_KEY, algorithm=ALGORITHM)
    return encoded_jwt

JWT로 만들 데이터를 입력으로 받아, 유효기간을 추가한 데이터로 토큰을 만들어주는 함수이다. 공식문서에서는 python-jose 라이브러리를 이용하였다.

토큰을 만들었으니 토큰을 해석(decode)하는 로직도 만들어보자. 사실 토큰은 그냥 BASE64이므로 해석이 가능하지만, secret key를 이용하여 검증하고, expire 시간이 유효한지 확인하는 로직도 포함한 것이다.

# auth.py
from fastapi.security import OAuth2PasswordBearer

# header에 있는 bearer(access token)값 읽어와 토큰 값을 제공한다.
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/login/")

async def get_token_data(token: Annotated[str, Depends(oauth2_scheme)])->dict:
    """
    token의 payload를 리턴한다
    만약 oauth2_scheme에서 토큰 정보 가져올 수 없다면 401 Not Authenticated를 리턴할 것임
    """
    credentials_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Could not validate credentials",
        headers={"WWW-Authenticate": "Bearer"},
    )
    try:
        payload = jwt.decode(token, ACCESS_TOKEN_SECRET_KEY,
                             algorithms=[ALGORITHM])
        return payload
    except JWTError:
        raise credentials_exception

jwt.decode()함수를 이용하여 토큰 정보를 얻어올 수 있다. oauth2_scheme을 통하여 bearer로 시작하는 Authorization header 값을 가져올 수 있으니, 이 의존성 함수를 주입해줌으로써 토큰 값을 이용할 수 있게 된다.

OAuth2PasswordBearer가 token_url과 함께 있는 것이 헷갈리는데, 아마 swagger 등을 통한 로그인 테스트시, lock 버튼으로 authorize인증을 간단하게 하도록 하기 위함으로 알면 편하다. 로그인까지 가면 이해할 수 있는데, 일단 login router url을 적어주면 된다고 생각하자.

로그인 및 토큰 발급(response)

이제 로그인 로직을 만들어주자. 로그인을 하기 위해서는 해당 username과 일치하는 사용자의 비밀번호와 입력된 비밀번호의 해시값이 같은지 확인해야 한다. 같다면 로그인이 성공하였으므로 access_token과 refresh_token을 발급한다.

  1. OAuth2PasswordBearer 초기화

    로그인시, 토큰이 발급되는 token_url을 입력해준다. 이 종속성함수를 이용하여 헤더에 있는 Bearer 토큰을 가져올 수 있으며, 해당 함수가 주입된 함수는 인증을 거쳐야 하는 함수로 간주된다.

  2. 로그인 api 만들기

    • 비밀번호 해싱

      회원가입시 비밀번호는 해싱되어 저장될 것이다(암호화하지 않으면 절대 안된다). 로그인 시에는 입력값과 DB값이 일치하는지 해싱을 통하여 검증하므로 이에 대한 context가 필요하다.

      # auth.py
from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

    • 로그인 함수 authenticate_user

      username, password를 입력받아 로그인을 처리하는 함수를 만들어주자.

      # auth.py
def verify_password(plain_password, hashed_password):
    """
    check password via pwd_context
    """
    return pwd_context.verify(plain_password, hashed_password)


def authenticate_user(username: str, password: str):
    """
    check if username and password is valid,
    or return False
    """
    # DB에서 회원 정보 가져오는 로직 필요
    userRow = {
        "username": "helloworld",
        # https://bcrypt-generator.com/에서 '123456' 해싱한 것
        "hashed_password": "$2a$12$f/tQdnxJCXsuwDkY.XzuJeCyatnkxsYvkXmSyGR04QxuC39ZEVs3y"
    }
    if userRow["username"] != username:
        return False
    if not verify_password(password, userRow["hashed_password"]):
        return False
    return userRow

      회원가입 로직과, user DB는 각자 구축하도록 하자…(?) 일단 나는 로그인이 성공하면 사용자 정보를 리턴하도록 하였다. 회원가입시, 비밀번호를 해싱하는 로직이 필요할 것인데, CryptContext를 이용할 수 있을 것이다. 사용자 정보를 가져왔다고 가정했을 때, verify_password 를 통하여 입력값과 원래 값이 일치하는지를 판별한다.

    • 로그인 api

      router를 달아주고, 이를 테스트해보자!

      # main.py
from typing import Annotated
from fastapi import FastAPI, Response, Depends, status, HTTPException
from fastapi.security import OAuth2PasswordRequestForm
from auth import authenticate_user, create_access_token, create_refresh_token

@app.post("/login/") # OAuth2PasswordBearer의 tokenUrl과 같아야 swagger에서 권한 테스트가 용이
async def login_for_access_token(
    response: Response,
    form_data: Annotated[OAuth2PasswordRequestForm, Depends()]
):
    user = authenticate_user(form_data.username, form_data.password)
    if not user:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Incorrect username or password",
            headers={"WWW-Authenticate": "Bearer"},
        )
    access_token = create_access_token(data={"sub": user["username"]})
    refresh_token = create_refresh_token(data={"sub": user["username"]})
    # refresh token은 httponly cookie로 저장
    response.set_cookie(key='token', value=refresh_token, httponly=True)

    return {
        "access_token": access_token,
        "token_type": "Bearer"
    }

      uvicorn main:app --reload로 실행한 후, http://localhost:8000/docs에 접근하여 login api를 테스트해보자!

      try it out!을 눌러 자동으로 생성된 Form을 통하여 username과 password를 입력해주자. Form은 자동으로 login 함수의 form_data 인자에 OAuth2PasswordRequestForm 의존성 주입을 해줬기에 가능하다. 로그인이 성공한다면 access_token이 리턴되고, refresh_token 역시 쿠키로 저장됨을 확인할 수 있다. (chrome 기준. F12 - Cookies - http://localhost:8000 에 들어가면 token이라는 key로 저장되어 있어야 함)

      틀린 값을 준다면, authenticate_user 함수에서 exception이 생겨 401 에러가 발생할 것이다. 이것도 테스트해보자!

토큰 해석 (decode) 및 인가(authorization)

이제는 클라이언트에서 발급 받은 토큰을 통하여 인가가 필요한 요청을 보낼 것이다. 우리는 이제 Authorization을 해주면 된다. Authorization은 access token으로만 수행하는 것이 우리 프로젝트의 목표였으니, 구현해보도록 하자.

  • 토큰 받아서 해석하기

    위에서 구현한 get_token_data를 이용하면 토큰의 데이터를 decode하고, 조작되지 않은 데이터라는 것을 알 수 있다.

    # main.py
@app.get("/payload/")
async def check_access_token(payload: Annotated[dict, Depends(get_token_data)]):
    """
    authenticated 되어있다면 payload data를 리턴
    아니라면 오류 발생
    """
    return {"data": payload}

    단순하게 인자로 dependency 함수를 넣어주면 해당 값을 함수에서 받아와서 사용할 수 있다. docs에서 테스트해보면, 오른쪽에 락 버튼이 있는데, 이를 이용하여 인증을 하고 bearer token을 자동으로 넣어주도록 테스트할 수 있다!(이를 위해 인증을 거쳐 토큰을 발급해주는 token_url을 유효하게 적어주는 것이 권장됨)

    토큰이 유효하다면 아래와 같이 payload 데이터를 확인할 수 있을 것이다.

    {
  "data": {
    "sub": "helloworld",
    "exp": 1697195035
  }
}

    만약 유효기간이 지났거나, payload가 조작되었다면 401 Couldn't validate ~~ 에러가 나올 것이고, token 값 자체가 없다면(로그인 되지 않은 상태) 401 Not Authenticated오류가 나와야할 것이다.

  • 사용자 식별하기

    payload값을 가져왔으니, 이를 통하여 사용자를 식별할 수 있다. access token의 payload에 있는 sub(subject) data를 가져와 DB에서 확인한다. DB 구조에 따라 해당 사용자가 접근이 가능한 권한(role or permission)인지 확인해볼 수도 있을 것이다!

  • Authorization

    보통 decode를 해서 sub을 가져올 수 있다면 이것을 의미한다.

    이 토큰의 데이터는 우리 서버에서 발급하여 조작되지 않고 아직 유효하다

    따라서, permission role (admin, super user, normal user 등)이 존재하는 서비스가 아니라면 그냥 인가를 해주면 되지만, payload의 데이터를 다시 확인하는 것이 보안적으로 권장되는 방법이다.

    DB가 없는 우리 서비스는 엇비슷하게 구현해볼 수 있다.

    # auth.py
async def is_payload_valid(payload : Annotated[dict, Depends(get_token_data)])->True:
    """
    payload가 유효한지 확인한다
    """
    invalid_exception = HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="This user is invalid",
    )
    # DB에서 payload['sub']에 해당하는 username이 있다면 유효한 사용자라고 판단
    validUserNameRow = [
        "helloworld",
        "hellowordl2", # ...
    ]
    if payload["sub"] in validUserNameRow:
        return True
    else:
        raise invalid_exception

    이제 이를 인증이 필요한 요청에 대하여 인자로 주입해주면 될 것이다!

    # main.py
@app.get("/authenticated/")
async def is_authenticated(valid : Annotated[bool, Depends(is_payload_valid)]):
    return {"data" : "authenticated"}

    valid하지 않은 사용자라면 is_payload_valid에서 This user is invalid 오류가 발생할 것이므로, 앞으로 인가가 필요한 모든 함수에 is_payload_valid를 이용하도록 하자.

    위의 validUserNameRow 배열을 바꿔가며 아래처럼 결과가 잘 나오나 확인하자!

refresh 토큰을 이용한 access token 재발급 (with RTR)

로그인시, refresh token을 발급하며 해당 refresh token을 추가하는 것으로 수정하자. 이는 전역 변수를 이용할 수 있다.

# auth.py
valid_refresh_list = []
def is_refresh_valid(token: str):
    global valid_refresh_list
    return token in valid_refresh_list


def add_refresh_token(token: str):
    global valid_refresh_list
    valid_refresh_list.append(token)


def remove_refresh_token(token: str):
    global valid_refresh_list
    try:
        valid_refresh_list.remove(token)
    except:
        # list에 해당 토큰이 없는 경우
        raise credentials_exception
...

def create_refresh_token(data: dict):
    """
    data : payload dictionary
    refresh token을 생성하고, 변수로 해당 토큰을 저장한다
    """
    to_encode = data.copy()
    expire = timedelta(minutes=REFRESH_TOKEN_EXPIRE_MINUTES) + \
        datetime.utcnow()
    to_encode.update({"exp": expire})
    encoded_jwt = jwt.encode(
        to_encode, REFRESH_TOKEN_SECRET_KEY, algorithm=ALGORITHM)
    add_refresh_token(encoded_jwt)  # refresh token에 추가한다
    return encoded_jwt

remove_refresh_token은 python 배열에 존재하지 않다면 에러가 발생되므로 이 경우에는 유효하지 않은 토큰 목록으로 간주하여 에러를 발생시키도록 한다.

이제 해당 refresh 토큰이 유효한지 확인하는 로직을 추가하자.

# auth.py
def verify_refresh_token(token: str) -> dict:
    """
    해당 refresh 토큰이 유효한지 확인
    유효하다면 payload return
    """
    try:
        payload = jwt.decode(
            token, REFRESH_TOKEN_SECRET_KEY, algorithms=[ALGORITHM])
        if is_refresh_valid(token):
            return payload
    except JWTError:
        raise credentials_exception

이를 이용하여 하나의 refresh token으로 하나의 access token만 생성이 가능하도록 구현할 수 있다!

# main.py
from typing import Union
from fastapi import Response, Cookie
from auth import verify_refresh_token, create_refresh_token, create_access_token

@app.post("/refresh/")
async def refresh(response: Response, token: Union[str, None] = Cookie('')):
    """
    http only cookie에서 refresh_token 값을 얻어와 검증하고, 유효하다면 새로운 토큰을 발급한다
    """
    prev_token = token
    payload = verify_refresh_token(prev_token)
    remove_refresh_token(prev_token)
    refresh_token = create_refresh_token(payload)
    access_token = create_access_token(payload)
    response.set_cookie(key='token', value=refresh_token, httponly=True)
    return {
        "access_token": access_token,
        "token_type": "Bearer"
    }

쿠키는 Cookie를 import하여 가져올 수 있으니 참고하자.! 다행히 과거에는 Swagger UI를 통한 쿠키 접근이 안됐다고 했던거 같은데, 지금은 가능한 것으로 보인다.

localhost:8000/docs에 접근하여 실행해보자. 서버 프로세스가 재시작 된 경우라면, global 변수 역시 초기화되므로 다시 로그인을 한 후에 진행하면 된다.

Execute를 누를때마다 token 값이 바뀌는 것을 확인할 수 있을 것이다.

token의 Value를 임의나 이전 토큰 값으로 수정하면 error가 나오는 것을 확인할 수 있을 것이다.

현실적으로 DB에 저장되야 하는 이유

사실, 런타임 변수로 발급한 refresh token을 관리하는 일은 비현실적인데, 다음과 같은 상황이 있기 때문이다

  1. 사용자 수가 많아 배열에 담을수록 메모리가 무거워진다

  2. 멀티 프로세싱 환경에서는 런타임 변수를 공유할 수 없다

    사실 이 방법은 fastapi의 경우 특정 상황에서의 해결 방법이 있다. 보통 gunicorn으로 uvicorn worker를 둬서 멀티 프로세싱을 처리하는데, 이 경우 process 간의 공유 변수(shared variable)을 이용하면 된다. 간단한 사용법은 다음과 같다.

    from multiprocessing import Manager

manager = Manager()
valid_refresh_list = manager.list()

def add_refresh_token(token : str):
    """
    global refresh token을 추가한다
    """
    global valid_refresh_list
    valid_refresh_list.apped(token)

    새로운 토큰이 발급될때마다 공유 변수에 추가해주는 방법이 있다… 하지만 역시 1번 문제가 발생하기 마련이다.

  3. 큰 서비스는 하나의 서버 컴퓨터에서 이루어지지 않는다

    런타임 변수를 어떻게든 공유하지 못하는 환경이 생길 수도 있다. 이 경우는 in memory DB를 써도 마찬가지로 발생하는 문제이다.

결론적으로, 안정적이고 큰 서비스를 운영해야하고 RTR 방식을 선택한다면 사용자 DB에 refresh같은 column을 만들어서 보관하자. 약간 빠르게 할 수 있는 방법으로는 (나만의 생각이긴 하지만) 인메모리 DB를 먼저 탐색하고 이후에 DB를 탐색하도록 refresh token 정보를 캐싱해두는 전략도 가능할 것 같다.

로그아웃 로직

refresh token을 지워줘서 access token을 더 이상 발급받지 못하도록 하면 된다!

물론 프론트에서도 새로고침을 하는 등으로 access token을 비워주는 작업을 하면 될 것이다.

# main.py
@app.post("/logout/")
async def logout(response: Response):
    response.delete_cookie(key='token', httponly=True)

요약

JWT에 대한 개념 자체가 익숙해졌다면 서버에서 인증 체계를 구현하는 것은 재미있는 것 같다! 유명한 비유로, 놀이공원 티켓을 발급한다고 검사하는 입장이라고 생각해보는 것이다…

  • 가장 쉬운 방법으로는 도장을 찍고 해당 도장이 있다면 통과시켜주는 것이겠다

  • 비교적 안전한 방법으로는 유효기간과 사용자 식별 정보가 포함된 정보를 조작이 불가하게 밀봉하고, 티켓(access token)을 확인하여 해당 사용자가 유효한지까지 확인하는 방법도 있을 것이다.

  • 또한, 누군가 버리고 간 티켓(refresh token)을 사용하지 못하도록, 놀이기구를 한번 탈때마다 새로운 티켓을 발급받아야 하도록(RTR) 한다고 생각해보자.
